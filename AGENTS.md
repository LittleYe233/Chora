# Chora Codebase Guide for Agents

This document provides essential information for AI agents working on the Chora codebase.

## 1. Build, Lint, and Test Commands

The project uses Gradle with Kotlin DSL. All commands should be run from the project root.

### Build
- **Assemble Debug APK:**
  ```bash
  ./gradlew assembleDebug
  ```
- **Assemble Release APK:**
  ```bash
  ./gradlew assembleRelease
  ```
- **Clean Build:**
  ```bash
  ./gradlew clean
  ```

### Linting & Formatting
- **Run Lint:**
  ```bash
  ./gradlew lint
  ```
  *Note: Check `app/build/reports/lint-results.xml` or `html` for output.*

### Testing
- **Run All Unit Tests:**
  ```bash
  ./gradlew test
  ```
- **Run All Instrumented Tests:**
  ```bash
  ./gradlew connectedAndroidTest
  ```
- **Run a Single Unit Test Class:**
  ```bash
  ./gradlew testDebugUnitTest --tests "com.craftworks.music.YourTestClass"
  ```
- **Run a Single Unit Test Method:**
  ```bash
  ./gradlew testDebugUnitTest --tests "com.craftworks.music.YourTestClass.testMethodName"
  ```

## 2. Code Style Guidelines

Follow these conventions to maintain consistency with the existing codebase.

### General
- **Language:** Kotlin (target JVM 21).
- **Frameworks:** Android Jetpack Compose, Hilt, Media3, Coil, Ktor.
- **Indentation:** 4 spaces.
- **Line Length:** Generally keep under 100-120 characters.
- **Braces:** Opening braces on the same line.

### Naming Conventions
- **Classes/Interfaces:** PascalCase (e.g., `LocalDataSource`, `MediaItem`).
- **Functions/Methods:** camelCase (e.g., `getLocalAlbums`, `createLocalPlaylist`).
- **Variables:** camelCase (e.g., `localProvider`, `playlistId`).
- **Constants:** SCREAMING_SNAKE_CASE (inside `companion object` or top-level).
- **Composables:** PascalCase (e.g., `SongList`, `PlayerScreen`).

### Imports
- Avoid wildcard imports (`import com.package.*`).
- Group imports: Android/AndroidX, Kotlin/Java, Third-party, Project-specific.
- Remove unused imports.

### Type Safety & Nullability
- Use explicit types when the type is not immediately obvious.
- Leverage Kotlin's null safety features (`?`, `?.`, `?:`).
- Avoid `!!` (double-bang) operator; use safe calls or `let` blocks.

### Dependency Injection (Hilt)
- Use Constructor Injection (`@Inject constructor(...)`) whenever possible.
- Annotate classes with `@Singleton` if they should be singletons.
- Use `@HiltViewModel` for ViewModels.
- Use `@AndroidEntryPoint` for Activities/Fragments/Composables (if needed).

### Coroutines & Flow
- Use `suspend` functions for I/O operations.
- Use `Flow` for reactive data streams (e.g., from Room or DataStore).
- Collect flows safely in UI (e.g., `collectAsStateWithLifecycle`).

### Error Handling
- Prefer returning `Result<T>` or nullable types (`T?`) over throwing exceptions for expected failures.
- Handle errors gracefully in the UI (show snackbars or error states).
- Log errors using a logger or `println` (if no logger is set up yet), but prefer a proper logging solution.

### Documentation
- Use KDoc (`/** ... */`) for public APIs and complex logic.
- Use `//` for implementation details.
- Mark unfinished work with `// TODO: Description`.

## 3. Project Structure
- `app/src/main/java/com/craftworks/music/`: Main source code.
  - `data/`: Data layer (DataSources, Models, Repositories).
  - `managers/`: Business logic and state managers.
  - `ui/`: Jetpack Compose UI components.
  - `providers/`: Data providers (Local, Navidrome, etc.).
- `app/src/main/res/`: Android resources (though Compose reduces reliance on XML).
- `build.gradle.kts`: Project and module level build configurations.

## 4. Key Libraries
- **UI:** Jetpack Compose (Material3).
- **Navigation:** Jetpack Navigation Compose.
- **Image Loading:** Coil.
- **Media:** AndroidX Media3 (ExoPlayer).
- **Networking:** Ktor.
- **DI:** Hilt.
- **Serialization:** Kotlinx Serialization.
